# Migration Status – VibeOverlay Studio

This document maps out the status of the database migration schema, cataloging existing objects, newly registered elements, skipped/idempotent states, and a schema drift assessment.

---

## 1. Catalog of Database Objects

### Existing Tables (v1 Schema Core)
These tables form the foundation of the database and were established during Phase 1:
* `profiles` (linked to `auth.users`)
* `projects` (user container rows)
* `scenes` (starting-soon, main-stream, etc.)
* `widget_styles` (basic styling fields)
* `widgets` (Zustand layout content registry)
* `scene_widgets` (layout placements junction)
* `settings` (stream settings parameters)
* `themes` (deprecated in favor of `theme_presets`)
* `scene_history` (local undo/redo snapstack)
* `templates` & `template_widgets` (for custom layouts)
* `assets` (media catalogs)

### Existing/Recreated RLS Policies
These Row Level Security (RLS) policies are dropped and recreated dynamically during migration:
* **`editor_preferences`**: `Users can CRUD editor preferences of their projects`
* **`asset_collections`**: `Users can CRUD asset collections of their projects`
* **`asset_tags`**: `Users can CRUD asset tags of their projects`
* **`asset_tag_map`**: `Users can CRUD asset tag associations`
* **`theme_presets`**: `Users can CRUD theme presets of their projects`
* **`animation_presets`**: `Users can CRUD animation presets of their projects`
* **`widget_groups`**: `Users can CRUD widget groups of their projects`

## 3. Migration Actions: New, Skipped, and Altered Objects

* **Newly Created Tables (if not exists)**:
  * `editor_preferences`
  * `asset_collections`
  * `asset_tags`
  * `asset_tag_map`
  * `theme_presets`
  * `animation_presets`
  * `widget_groups`
* **Safely Altered / Column Shifts**:
  * `assets`: Added column `collection_id` uuid (if not already present).
  * `scene_widgets`: Dropped `scene_widgets_parent_id_fkey` constraint and altered `parent_id` column type to `text` to resolve folder string assignment warnings.
* **Skipped Memberships**:
  * Any tables already registered inside the `supabase_realtime` publication catalog are automatically ignored by our catalog query loop to bypass duplicate member errors.

---

## 4. Schema Drift & Consistencies Comparison

The live database layout was compared against the expected schema configurations:
1. **`parent_id` uuid conflict**:
   * *Status*: Resolved. The database column `parent_id` on `scene_widgets` was converted to `text`, matching the virtual grouping ID generated by the frontend.
2. **Missing Realtime replication**:
   * *Status*: Resolved. Settings, widgets, and layout tables are successfully registered in the realtime broadcast loop.
3. **Policies collision**:
   * *Status*: Resolved. All RLS policies are dropped prior to instantiation, allowing developer re-runs of the SQL file without `ERROR: policy already exists` failure halts.
